Conceptual Deep Dive

Mental model komprehensif dari seluruh topik yang diuji dalam UTS ini.

CIA Triad — Fondasi Keamanan Informasi

CIA Triad adalah tiga pilar utama yang menjadi dasar dari setiap sistem keamanan. Setiap keputusan desain keamanan harus mempertimbangkan ketiga aspek ini.

C — Confidentiality

Data hanya boleh diakses oleh pihak yang berwenang. Dicapai melalui enkripsi, access control, dan authentication.

I — Integrity

Data tidak boleh dimodifikasi secara tidak sah. Dijaga dengan hashing, digital signature, dan checksums.

A — Availability

Sistem harus selalu tersedia saat dibutuhkan. Ancaman: DoS/DDoS attacks, hardware failure.

Secure SDLC — Keamanan di Setiap Fase

Keamanan bukan ditambahkan di akhir, tapi diintegrasikan sejak fase pertama. Inilah inti dari Secure Software Development Life Cycle (SSDLC).

1
Requirements — Tentukan security requirements sejak awal
2
Design — Threat modeling & secure architecture (TAHAP PALING PENTING)
3
Implementation — Secure coding practices, input validation
4
Testing — Penetration testing, SAST/DAST, code review
5
Deployment & Maintenance — Security patching, monitoring, audit

InjectionSQL Injection

ScriptXSS (Cross-Site Scripting)

Penyerang menyisipkan skrip berbahaya di halaman web yang kemudian dieksekusi di browser korban.

NetworkMan-in-the-Middle

Penyerang mencegat komunikasi antara dua pihak tanpa sepengetahuan mereka untuk mengintip atau memanipulasi data.

AuthBrute Force

Menebak password secara otomatis dengan mencoba semua kemungkinan kombinasi hingga menemukan yang benar.

SessionSession Hijacking

Penyerang mencuri session token yang valid untuk mengambil alih sesi login pengguna yang sedang aktif.

HumanSocial Engineering

Memanfaatkan kelemahan manusia melalui manipulasi psikologis, bukan kelemahan teknis sistem.

Common Pitfalls — Kesalahan Umum Mahasiswa

Hashing ≠ Enkripsi. Hashing adalah transformasi SATU ARAH (tidak bisa dikembalikan). Enkripsi adalah dua arah (bisa didekripsi dengan kunci).

SSDLC dimulai dari fase Desain, bukan Testing atau Deployment. Keamanan harus dipikirkan SEBELUM menulis kode.

Enkripsi Simetris = 1 kunci untuk enkripsi DAN dekripsi. Asimetris = pasangan kunci publik + privat.

Social Engineering menyerang manusianya, bukan sistemnya — inilah yang membedakannya dari serangan teknis lain.

Visual Studio Code adalah code editor biasa, BUKAN security testing tool seperti SonarQube, Fortify, atau OWASP ZAP.

Algorithm & Logic Visualization

Password Hashing + Salting

Proses penyimpanan password yang aman. Tanpa salting, dua pengguna dengan password sama akan memiliki hash yang sama — berbahaya!

// BURUK — menyimpan password plaintext ❌
function savePasswordUnsafe(password) {
    database.save(password);  // JANGAN PERNAH!
}

// BURUK — hashing tanpa salt ❌
function savePasswordNoSalt(password) {
    const hash = sha256(password);
    database.save(hash);  // rentan rainbow table attack
}

// BENAR — hashing dengan salt ✅
function savePasswordSafe(password) {
    const salt = generateRandomString(16);  // salt acak per-user
    const saltedPassword = password + salt;
    const hash = bcrypt(saltedPassword);        // hash yang kuat
    database.save({ hash, salt });              // simpan keduanya
}

// Verifikasi password saat login
function verifyPassword(inputPassword, storedHash, storedSalt) {
    const hash = bcrypt(inputPassword + storedSalt);
    return hash === storedHash;  // compare hash, bukan password
}
STEP 1 — Generate Salt

Salt adalah string acak yang unik untuk setiap pengguna. Ini memastikan dua user dengan password yang sama pun memiliki hash berbeda.

STEP 2 — Gabungkan Password + Salt

Password asli digabungkan dengan salt sebelum di-hash, sehingga hasil akhirnya unik per pengguna.

STEP 3 — Hashing (One-Way)

Fungsi hash (bcrypt, sha256) mengubah data menjadi bentuk tetap yang TIDAK BISA dikembalikan. Inilah yang membedakan hash dari enkripsi.

STEP 4 — Simpan Hash + Salt

Simpan keduanya di database. Saat login, proses diulang: input + salt → hash, lalu bandingkan hasilnya.

SQL Injection — Serangan & Mitigasi

Terjadi ketika input pengguna tidak divalidasi dan langsung disisipkan ke query SQL, memungkinkan penyerang memanipulasi query database.

// VULNERABLE — rentan SQL Injection ❌
function loginVulnerable(username, password) {
    const query = `SELECT * FROM users 
                   WHERE username='${username}' 
                   AND password='${password}'`;
    // Jika username = "admin' --"
    // Query jadi: WHERE username='admin' --' AND password='...'
    // Bagian password di-comment, login berhasil tanpa password!
    return database.execute(query);
}

// AMAN — menggunakan Prepared Statement ✅
function loginSafe(username, password) {
    const query = "SELECT * FROM users WHERE username=? AND password=?";
    return database.prepare(query).execute(username, password);
    // Input diperlakukan sebagai DATA, bukan bagian dari query
}
PENYEBAB

Input pengguna tidak divalidasi dan langsung digunakan dalam query database tanpa sanitasi.

SOLUSI

Gunakan Prepared Statements / Parameterized Queries. Selalu validasi dan sanitasi input pengguna.

Simetris vs Asimetris Encryption

// ENKRIPSI SIMETRIS — 1 kunci untuk enkripsi & dekripsi
const secretKey = "kunci-rahasia-sama";

const encrypted = AES.encrypt(data, secretKey);   // enkripsi
const decrypted = AES.decrypt(encrypted, secretKey); // dekripsi, kunci SAMA
// Contoh: AES, DES, 3DES — cepat, bagus untuk data besar

// ─────────────────────────────────────────────────────────

// ENKRIPSI ASIMETRIS — 2 kunci: public + private
const { publicKey, privateKey } = RSA.generateKeyPair();

const encrypted = RSA.encrypt(data, publicKey);     // enkripsi dgn public key
const decrypted = RSA.decrypt(encrypted, privateKey); // dekripsi dgn private key
// Contoh: RSA, ECC — lambat, bagus untuk key exchange
Simetris: 1 kunci yang sama digunakan untuk enkripsi dan dekripsi (lebih cepat)
Asimetris: pasangan kunci publik (untuk enkripsi) dan privat (untuk dekripsi)
HTTPS menggunakan keduanya: asimetris untuk key exchange, simetris untuk data transfer
