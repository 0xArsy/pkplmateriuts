Conceptual Deep Dive

Quiz 1

Security Principles (Saltzer & Schroeder)

8 prinsip desain keamanan klasik yang menjadi fondasi semua sistem aman.

Least Privilege

Beri akses minimum sesuai kebutuhan tugas. Mengurangi exposure sistem.

Fail-safe Defaults

Default = TOLAK akses. Jika sistem gagal, ia harus gagal secara aman (deny, bukan grant).

Economy of Mechanism

Mekanisme keamanan harus sesederhana mungkin. Kompleksitas → lebih banyak bug.

Defense in Depth

Berlapis-lapis pertahanan. Jika satu lapisan tembus, masih ada lapisan berikutnya.

Separation of Privilege

Akses kritis butuh lebih dari satu kondisi / otorisasi (misal: dual approval).

Psychological Acceptability

Kontrol keamanan harus mudah digunakan, agar pengguna tidak mencari cara "bypass".

Quiz 2

UML, OCL & Secure Functional Model

<<include>>

Use case SELALU menyertakan use case lain. Wajib, bukan opsional.

<<extend>>

Use case KONDISIONAL — hanya dieksekusi dalam kondisi tertentu.

Composition vs Aggregation

Composition: bagian BERGANTUNG pada keseluruhan. Aggregation: bagian dapat EKSIS sendiri.

OCL — Object Constraint Language

OCL digunakan untuk mendefinisikan precondition (syarat sebelum operasi) dan postcondition (jaminan setelah operasi). Keyword self merujuk ke objek dalam konteks saat ini. Stereotype adalah anotasi semantik saja — tidak menjamin keamanan otomatis!

Quiz 3

Secure Dynamic Model — Pre/Postcondition & Sequence Diagram

Aturan Precondition & Postcondition
  • ✦ Jika precondition gagal → operasi DITOLAK, postcondition tidak dievaluasi
  • ✦ Jika precondition OK tapi postcondition dilanggar → bug implementasi
  • ✦ Operasi tanpa postcondition = tidak ada jaminan state setelah eksekusi
  • ✦ Stereotype bersifat anotasi — OCL TIDAK otomatis mengikuti stereotype
Elemen Sequence Diagram
  • Activation bar = durasi objek memproses pesan
  • Destroy message = simbol X pada lifeline
  • Fork node = eksekusi paralel beberapa aktivitas
  • Join node = banyak panah masuk ke horizontal bar
  • Final node = lingkaran dengan titik di dalamnya
  • Self-call rekursif = loopback arrow + activation bar tambahan

DREAD Scoring

Damage + Reproducibility + Exploitability + Affected Users + Discoverability. Skor lebih tinggi = prioritas mitigasi lebih tinggi.

Quiz 5

Authentication, Authorization & OAuth

What you KNOW

Password, PIN, jawaban pertanyaan keamanan

What you HAVE

Kartu ATM, token OTP, smartphone (Google prompt memilih angka = HAVE)

What you ARE

Fingerprint, face ID, retina scan. CEK KTM di ujian = ARE (identitas fisik)

⚠ Common Pitfalls — Quiz 5
• OAuth melakukan KEDUANYA — autentikasi DAN otorisasi (bukan otorisasi saja!)
• MFA yang benar: harus kombinasi DIFFERENT factors. PIN + password = sama-sama "what you know" → BUKAN MFA
• Cek KTM = "What you are" (verifikasi identitas fisik mahasiswa), bukan "what you have"
• Password terbaik = yang paling PANJANG (bukan paling kompleks kombinasi)
• SCELE enroll kuliah = autentikasi SAJA (login dulu, tidak ada otorisasi tambahan untuk enroll)
• Aplikasi aman = SALAH SEMUA dari pilihan A, B, C, D yang ada — harus lebih dari sekadar auth+authz
Quiz 6

Input Validation & Sanitization

Validation vs Sanitization

Validation: memeriksa apakah data dapat DITERIMA (ya/tidak). Sanitization: mengubah data menjadi bentuk yang AMAN.

Syntactic vs Semantic Validation

Syntactic: format (email harus ada @, nomor harus angka). Semantic: logika bisnis (tanggal mulai harus sebelum selesai, umur tidak bisa 250 tahun). Syntactic DULU baru Semantic.

Whitelisting vs Blacklisting

Whitelisting (LEBIH BAIK): hanya izinkan yang diketahui aman. Blacklisting: blokir yang diketahui berbahaya (mudah di-bypass).

Stack Canaries (Buffer Overflow Defense)

Nilai acak yang ditempatkan sebelum return address. Jika nilai berubah (karena overflow), program DIHENTIKAN.

Sanitize as close to the SINK

Sanitasi dilakukan tepat SEBELUM data digunakan pada operasi berbahaya (bukan saat input masuk).

Client-side validation saja TIDAK CUKUP

Bisa di-bypass dengan memodifikasi request langsung ke server. Selalu validasi di server-side juga.

Quiz 7

Standard Web Application Vulnerabilities

XSS (Cross-Site Scripting)

Terjadi karena menampilkan input tanpa output encoding. Mitigasi: htmlspecialchars(), CSP header.

CSRF

Browser otomatis kirim cookie korban ke situs target. Mitigasi: Anti-CSRF token unik per session.

Session Fixation

Penyerang menetapkan session ID tertentu SEBELUM korban login, lalu menggunakannya setelah login.

Credential Stuffing

Memanfaatkan reuse password. Data bocor dari layanan A dipakai di layanan B. Mitigasi: MFA.

Clickjacking

Halaman diload dalam iframe tersembunyi. Mitigasi: X-Frame-Options: DENY.

HSTS

HTTP Strict Transport Security — mencegah protocol downgrade attack & MitM dengan memaksa HTTPS.

Key Terms Wajib Hafal
Accountability — audit trail, merekam siapa melakukan apa
Integrity — data tidak dapat diubah tanpa izin
Trust Boundary — perubahan tingkat kepercayaan dalam DFD
Threat Agent — entitas yang punya kemampuan & niat eksploitasi
Exposure vs Loss — terpapar ancaman vs kerugian aktual
Non-functional req. — performa (1000 transaksi/mnt), bukan fitur
Parameterized Query — input diperlakukan sebagai data, bukan SQL
Content-Security-Policy — daftar sumber tepercaya untuk konten browser
Common Pitfalls Lintas Quiz

Audit trail ≠ Integrity — audit trail berkaitan dengan Accountability, bukan Integrity

Stereotype ≠ keamanan — stereotype UML hanya label semantik, tidak enforce keamanan

Precondition gagal → TOLAK operasi total — bukan jalankan dulu baru cek

MFA ≠ dua password — harus kombinasi faktor berbeda (know/have/are)

DDoS ≠ Injection — DDoS adalah availability attack, bukan injection

CSRF ≠ XSS — CSRF memanfaatkan cookie otomatis browser; XSS menyisipkan skrip

Algorithm & Logic Visualization

OCL Precondition / Postcondition — Cara Baca & Analisis

Quiz 3 sering menguji kemampuan membaca OCL dan menentukan apakah precondition atau postcondition yang dilanggar.

// Contoh OCL dari Quiz 3
context Person::createLogin(username : String, password : String)
pre:  username.size() >= 8 and password.size() >= 12
post: self.isAuthenticated = true

// Skenario: password hanya 6 karakter, tapi sistem set isAuthenticated = true
// ANALISIS:
// 1. Apakah precondition dilanggar? YA — password.size() = 6 < 12
// 2. Karena precondition gagal → operasi SEHARUSNYA ditolak
// 3. Tapi sistem tetap set isAuthenticated = true → postcondition juga dilanggar
// JAWABAN: Pelanggaran KEDUANYA (precondition DAN postcondition)
// ❌ Salah: "hanya precondition" — postcondition juga dilanggar karena
//    sistem seharusnya tidak berjalan tapi tetap mengubah state
STEP 1 — Cek Precondition

Apakah semua kondisi sebelum method dipanggil terpenuhi? Jika tidak → operasi DITOLAK total.

STEP 2 — Jalankan Operasi

Jika precondition OK, operasi boleh berjalan. Perhatikan apakah ada perubahan state yang relevan.

STEP 3 — Verifikasi Postcondition

Setelah operasi selesai, apakah state sesuai yang dijanjikan? Jika tidak → bug implementasi.

Attack Tree — OR node vs AND node

Kunci: "salah satu cukup" = OR node | "semua harus terpenuhi" = AND node

// OR node — salah satu cabang cukup untuk mencapai root goal
Root: "Ambil alih akun"
├─ [OR]─ Menebak password (brute force)
├─ [OR]─ Phishing pengguna
└─ [OR]─ Reset password via email yang dikompromikan
// → Penyerang hanya perlu berhasil di SATU dari tiga cara

// AND node — SEMUA langkah harus dilakukan (sequential)
Root: "Ambil alih sistem admin"
├─ [AND]─ STEP 1: Dapatkan kredensial admin
└─ [AND]─ STEP 2: Login ke panel admin
// → Penyerang HARUS berhasil di kedua langkah

// Cara cepat bedain:
// "dapat dilakukan secara independen" → OR node
// "diperlukan" / "harus" / "langkah berurutan" → AND node

SQL Injection — Anatomy of a Payload

-- Payload: ' OR '1'='1 dimasukkan ke field email
-- Query asli:
SELECT * FROM users WHERE email='[INPUT]' AND password='[PASS]'

-- Setelah injeksi:
SELECT * FROM users WHERE email='' OR '1'='1' AND password='...'

-- Parsing oleh database:
-- WHERE email='' OR '1'='1'  ← '1'='1' selalu TRUE!
-- Kondisi WHERE menjadi selalu TRUE → login berhasil tanpa password

-- DAMPAK: Integrity + Confidentiality (CIA)
-- MITIGASI: Parameterized queries
const q = "SELECT * FROM users WHERE email=? AND password=?";
db.prepare(q).execute(email, password); // input = DATA, bukan SQL
