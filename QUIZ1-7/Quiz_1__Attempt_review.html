Conceptual Deep Dive

Extended Security Goals

Di luar CIA Triad, ada security goals tambahan yang sering diuji.

COMMON PITFALL

Audit trail sering dikira berkaitan dengan Integrity. Padahal audit trail mencatat SIAPA melakukan APA dan KAPAN — ini adalah Accountability. Integrity = data tidak berubah tanpa izin.

Prinsip Keamanan Saltzer & Schroeder (8 Prinsip)

Prinsip-prinsip klasik yang diajukan 1975, masih relevan sampai sekarang. Semua keluar di kuis!

COMMON PITFALL

Mengurangi endpoint API publik bukan "meningkatkan availability" — tujuannya adalah mengurangi exposure (attack surface). Semakin sedikit endpoint terbuka, semakin kecil permukaan serangan.

Konsep Ancaman: Threat, Vulnerability, Risk, Exposure, Loss

KEY TERM

Threat Agent = entitas yang MEMILIKI kemampuan DAN NIAT untuk mengeksploitasi kerentanan. Berbeda dari threat (ancaman abstrak) dan vulnerability (kelemahan sistem).

Ransomware vs CIA Triad

Ransomware mengenkripsi file → menyerang Availability (file tidak bisa diakses) dan Integrity (file dimodifikasi tanpa izin). BUKAN hanya Confidentiality.

Confidentiality

Data bocor ke pihak tidak berwenang. Ransomware biasanya TIDAK membocorkan (kecuali double extortion).

Integrity ✓

File dienkripsi = diubah tanpa izin. Melanggar Integrity.

Availability ✓

File tidak bisa dibuka tanpa kunci. Melanggar Availability.

Algorithm & Logic Visualization

Fail-Safe Defaults — Logika Keamanan

// ❌ MELANGGAR Fail-Safe Defaults
function checkAccess(user) {
  try {
    if (authService.verify(user)) {
      return "GRANTED";
    }
  } catch (err) {
    // Sistem crash → akses diberikan penuh! 🚨
    return "GRANTED";
  }
}

// ✅ MENGIKUTI Fail-Safe Defaults
function checkAccess(user) {
  try {
    if (authService.verify(user)) {
      return "GRANTED";
    }
    return "DENIED"; // default = tolak
  } catch (err) {
    // Error → TOLAK akses (aman)
    return "DENIED";
  }
}
STEP-BY-STEP

1️⃣ Sistem mencoba verifikasi user

2️⃣ Jika verifikasi berhasil → izinkan akses

3️⃣ Jika verifikasi gagal → tolak akses

4️⃣ Jika terjadi ERROR (crash) → TETAP TOLAK (fail secure)

⚠️ Prinsip: saat gagal, sistem harus ke kondisi PALING AMAN (denied), bukan paling permisif (granted)

Defense in Depth — Segmentasi Jaringan

// Tanpa Defense in Depth (satu lapis)
Internet ──→ [Firewall] ──→ Public Server ──→ Database
                               ↑ jika server dikompromikan,
                                 database langsung bisa diakses!

// Dengan Defense in Depth (berlapis)
Internet ──→ [Firewall] ──→ [DMZ] ──→ Public Server
                                          │
                                    [Firewall 2]
                                          │
                                   [Internal Network]
                                          │
                                      Database
// Penyerang harus menembus SETIAP lapisan

Soal Quiz: "tidak ada segmentasi jaringan antara server publik dan database internal" → melanggar Defense in Depth. Segmentasi = lapisan perlindungan tambahan.

Session Management — Mencegah Session Hijacking

// ❌ Praktik buruk
session_id = md5(username + timestamp)  // mudah ditebak
GET /dashboard?session=abc123           // ID di URL → bocor di log!
Set-Cookie: session=abc123               // tanpa Secure & HttpOnly

// ✅ Praktik aman
session_id = crypto.randomBytes(32)     // acak & panjang (256-bit)
Set-Cookie: session=...; Secure; HttpOnly; SameSite=Strict
// Secure → hanya lewat HTTPS
// HttpOnly → tidak bisa diakses JavaScript (anti-XSS)
// SameSite → mencegah CSRF attack
