Conceptual Deep Dive

Relasi Use Case: <<include>> vs <<extend>>

<<include>>
  • • Use case SELALU menyertakan use case lain
  • • Wajib, tidak kondisional
  • • Contoh: Login → <<include>> → Verifikasi Kredensial
  • • Setiap kali Login dipanggil, Verifikasi Kredensial PASTI dipanggil
<<extend>>
  • • Use case bersifat TAMBAHAN dan KONDISIONAL
  • • Opsional, hanya dalam kondisi tertentu
  • • Contoh: Login → <<extend>> → Tampilkan Peringatan Login Mencurigakan
  • • Hanya muncul jika ada kondisi khusus (login dari IP asing)
COMMON PITFALL

Ingat: <<include>> = SELALU (mandatory). <<extend>> = KADANG-KADANG (kondisional). Banyak yang terbalik!

Class Diagram: Aggregation vs Composition

Aggregation (◇ diamond kosong)
  • • Hubungan "has-a" LEMAH
  • • Bagian DAPAT eksis tanpa keseluruhan
  • • Siklus hidup independen
  • • Contoh: Universitas ◇── Mahasiswa (mahasiswa bisa ada tanpa universitas)
  • • Contoh: Tim ◇── Pemain (pemain bisa pindah tim)
Composition (◆ diamond hitam)
  • • Hubungan "has-a" KUAT
  • • Bagian SANGAT bergantung pada keseluruhan
  • • Siklus hidup terikat — jika whole dihapus, part ikut dihapus
  • • Contoh: Rumah ◆── Ruangan (ruangan tidak ada tanpa rumah)
  • • Contoh: Order ◆── OrderItem
COMMON PITFALL

Mnemonic: Aggregation = bagian bisa "pergi" sendiri. Composition = bagian "mati" bersama keseluruhan. Ingat dengan analogi: Pemain (Aggregation) vs Ruangan dalam Gedung (Composition).

UML Security Stereotype

Stereotype ditulis dalam format <<namaStereotype>>. Tanpa constraint OCL tambahan, stereotype hanya bersifat label semantik — tidak otomatis mengubah perilaku sistem!

KEY TERM

Stereotype TIDAK eksekutabel sendiri. <<confidential>> tidak otomatis mengenkripsi message. Harus ada implementasi nyata (mekanisme enkripsi, constraint OCL) yang menyertainya.

OCL (Object Constraint Language)

Bahasa formal untuk mendefinisikan constraint dalam model UML. self merujuk pada objek dalam konteks saat ini (bukan semua class, bukan database).

pre: (Precondition)

Kondisi yang HARUS terpenuhi SEBELUM operasi dijalankan. Jika tidak terpenuhi → operasi DITOLAK.

post: (Postcondition)

Kondisi yang HARUS benar SETELAH operasi berhasil dijalankan. Jika tidak → postcondition dilanggar.

Misuse Case & Non-Functional Requirements

Misuse Case Modeling
  • • Memodelkan bagaimana penyerang bisa menyalahgunakan sistem
  • • Threat-driven modeling = mulai dari ancaman dan tujuan penyerang
  • • Countermeasure = use case yang MENGURANGI atau MENCEGAH ancaman
  • • Sifat secure functional model = PREVENTIF (bukan reaktif)
Non-Functional Requirement (NFR)
  • • NFR = bagaimana sistem melakukan sesuatu (kualitas/atribut)
  • • Functional req = apa yang sistem lakukan (fitur)
  • • Contoh NFR performa: "sistem memproses 1000 transaksi/menit"
  • • Contoh NFR keamanan: "password harus min 12 karakter"

Algorithm & Logic Visualization

<<include>> vs <<extend>> — Use Case Diagram

// Use Case: Sistem Ticketing

// ─── <> ─── (SELALU dijalankan)
[Beli Tiket] ──<<include>>──> [Verifikasi Pembayaran]
// Setiap kali "Beli Tiket" dijalankan, "Verifikasi Pembayaran" PASTI dijalankan
// Tidak ada pilihan - ini adalah bagian wajib dari alur

// ─── <> ─── (KONDISIONAL, hanya jika ada kondisi tertentu)
[Beli Tiket] ◀──<<extend>>── [Tampilkan Peringatan Stok Habis]
// "Tampilkan Peringatan" hanya muncul JIKA stok tiket kurang dari 5
// Tanpa kondisi itu, alur berjalan normal tanpa peringatan

// Analogi sederhana:
// <> = setiap BELI selalu ada BAYAR (wajib)
// <> = kadang ada DISKON jika punya promo code (opsional)
STEP-BY-STEP: Login <<include>> Verifikasi Kredensial

1️⃣ User mengklik tombol Login

2️⃣ Sistem OTOMATIS memanggil "Verifikasi Kredensial" (karena <<include>>)

3️⃣ Tidak ada cara login tanpa melalui verifikasi — ini SELALU terjadi

4️⃣ Jika verifikasi berhasil → user masuk. Jika gagal → tolak

OCL — Cara Membaca & Menganalisis

// Struktur OCL:
context KlasNama::namaMetode(params)
pre:  kondisi_yang_harus_benar_SEBELUM_operasi
post: kondisi_yang_harus_benar_SESUDAH_operasi

// Contoh dari Quiz:
context Order::addPayment(amount : Real)
pre:  amount > 0 and self.status = 'Pending'
post: self.totalPaid = self.totalPaid@pre + amount

// ANALISIS: Jika dipanggil saat status = 'Confirmed'
// pre check: self.status = 'Confirmed' ≠ 'Pending' → FALSE!
// → Operasi DITOLAK (precondition gagal)
// → Postcondition tidak dievaluasi sama sekali

// Kata kunci: @pre = nilai atribut SEBELUM operasi dijalankan
// self.totalPaid@pre = nilai totalPaid sebelum method ini jalan

Aggregation vs Composition — Kode & Diagram

// AGGREGATION (◇) — "has-a" lemah, bagian bisa eksis sendiri
class Team {
  private players: Player[];  // Player ada sebelum Team
  addPlayer(p: Player) { ... }
  removePlayer(p: Player) { ... } // Player tetap ada setelah diremove
}
class Player { // Player bisa ada tanpa Team!
  name: string;
}

// COMPOSITION (◆) — "has-a" kuat, bagian mati bersama whole
class Order {
  private items: OrderItem[] = [];
  constructor() {
    // OrderItem dibuat DI DALAM Order
    this.items.push(new OrderItem(...));
  }
  // Jika Order dihapus → OrderItem IKUT dihapus!
}

// Diagram UML:
// Aggregation: Team ◇─────── Player  (diamond kosong)
// Composition: Order ◆─────── OrderItem  (diamond hitam)
