🧠 Conceptual Deep Dive – Threat Modeling

🔑 STRIDE Framework

S – Spoofing: memalsukan identitas
T – Tampering: memodifikasi data
R – Repudiation: menyangkal tindakan
I – Information Disclosure: bocornya info
D – Denial of Service: ganggu ketersediaan
E – Elevation of Privilege: naikkan hak akses

🌳 Attack Tree Nodes

OR node: salah satu cabang cukup untuk capai goal
AND node: SEMUA cabang harus terpenuhi
Root node: goal utama penyerang (paling atas)
Leaf node: aksi primitif terbawah (tidak punya anak)

⚠️ Common Pitfalls

  • Leaf vs OR node: "Leaf" = simpul daun (tidak punya cabang lagi), bukan berarti "satu cabang cukup"
  • Spoofing vs EoP: Spoofing = memalsukan IDENTITAS. EoP = mendapat hak lebih tinggi dari seharusnya
  • Tampering vs Info Disclosure: Tampering = UBAH data, Info Disclosure = BOCOR/BACA data tanpa izin
  • DREAD: Damage, Reproducibility, Exploitability, Affected Users, Discoverability. Skor lebih tinggi = prioritas lebih tinggi

📊 DREAD Scoring Mental Model

// DREAD = rata-rata 5 dimensi (skala 1–10)
DREAD_score = (Damage + Reproducibility + Exploitability 
             + Affected_users + Discoverability) / 5

Threat A: (8+8+7+9+7) / 5 = 7.8  ← Prioritas TINGGI
Threat B: (4+4+3+5+4) / 5 = 4.0  ← Prioritas rendah

🗺️ DFD Components & Trust Boundaries

// Komponen DFD dan fungsinya dalam Threat Modeling

[External Entity]  ──trust boundary──  [Process]  ──  [Data Store]
  (Customer)                          (Web Server)    (Database)
      │                                    │
      │         ANCAMAN DI SETIAP KOMPONEN:
      │
  External Entity  →  Spoofing, Repudiation
  Process          →  Tampering, EoP, Info Disclosure
  Data Store       →  Tampering, Info Disclosure, Repudiation
  Data Flow        →  Tampering, Info Disclosure
  Trust Boundary   →  garis batas kepercayaan antar zona

// Trust Boundary WAJIB ada antara:
  - Internal system ↔ External API / Gateway
  - User browser   ↔ Web Server
  - Different privilege levels

🧠 Conceptual Deep Dive – Auth & MFA

💭 What You Know

Password, PIN, security question. Bisa dicuri/lupa.

📱 What You Have

Kartu ATM, HP (OTP), token hardware, KTM*. Bisa hilang/dicuri.

🧬 What You Are

Fingerprint, face ID, iris scan. Tidak bisa diganti jika bocor.

⚠️ MFA = faktor dari kategori BERBEDA. PIN + Password = 2× "what you know" = bukan MFA. Kartu ATM + OTP via HP = 2× "what you have" = bukan MFA sejati menurut definisi ketat. 
// Authentication vs Authorization

Authentication = "Siapa kamu?" → verifikasi identitas
  Contoh: login dengan username + password

Authorization  = "Apa yang boleh kamu lakukan?" → kontrol akses  
  Contoh: mahasiswa bisa lihat nilai sendiri, tapi tidak bisa edit

OAuth 2.0      = framework untuk Authorization
OIDC           = extension OAuth untuk Authentication juga
→ Dalam praktik modern, OAuth melakukan keduanya

🧠 Conceptual Deep Dive – Input Validation

🔑 Validation vs Sanitization

Validation: memeriksa – accept or reject
Sanitization: mengubah – clean/escape/transform
Keduanya diperlukan secara bersamaan!

🔑 Syntactic vs Semantic

Syntactic: format benar? (email punya @, tanggal dd/mm/yyyy)
Semantic: nilai masuk akal? (umur < 150, tanggal mulai < selesai)
Syntactic DULU, baru Semantic!
// Parameterized Query vs String Concatenation

// ❌ RENTAN SQLi (string concat)
query = "SELECT * FROM users WHERE email = '" + email + "'"
// Input: ' OR '1'='1  → auth bypass!

// ✅ AMAN (parameterized)
query = "SELECT * FROM users WHERE email = ?"
execute(query, [email])
// Input diperlakukan sebagai DATA, bukan kode SQL
⚠️ Common Pitfalls
  • Client-side validation mudah di-bypass via Burp Suite / curl → WAJIB validasi server-side juga
  • Whitelisting > Blacklisting: lebih aman izinkan yang diketahui aman daripada blokir yang diketahui berbahaya
  • "Sanitize as close to sink" = sanitize RIGHT BEFORE use, bukan saat input masuk (karena data bisa melewati banyak layer)

🧠 Conceptual Deep Dive – Web Vulnerabilities

🔑 XSS vs CSRF

XSS: inject script ke halaman → berjalan di browser korban
CSRF: buat browser korban kirim request ke situs lain
XSS = "attacker berbicara ke browser", CSRF = "attacker memakai browser korban"

🔑 Security Headers

CSP: whitelist sumber script/style
HSTS: paksa HTTPS, cegah downgrade
X-Frame-Options: DENY: cegah clickjacking
HttpOnly cookie: JS tidak bisa baca cookie
// SQL Injection: ' OR '1'='1 mechanism
// Query original:
SELECT * FROM users WHERE email = '[INPUT]' AND password = '...'

// Setelah payload ' OR '1'='1 dimasukkan sebagai email:
SELECT * FROM users WHERE email = '' OR '1'='1' AND password='...'
// '1'='1' selalu TRUE → WHERE clause selalu TRUE → semua user dikembalikan
// → login berhasil tanpa password yang benar!
// Cookie Security Flags
Set-Cookie: sessionId=abc123;
  Secure;      // hanya kirim via HTTPS
  HttpOnly;    // JS tidak bisa baca (document.cookie) → cegah XSS steal cookie
  SameSite=Strict // tidak dikirim pada cross-site request → cegah CSRF

STRIDE Quiz 4

Seorang pengguna biasa berhasil mengakses fitur admin dan menghapus data user lain karena tidak ada pengecekan role. Ancaman STRIDE apa yang terjadi?

✅ Jawaban: Elevation of Privilege

EoP terjadi ketika seseorang mendapatkan akses/kemampuan LEBIH dari yang seharusnya dimiliki. Dalam kasus ini, user biasa dapat melakukan aksi admin → hak aksesnya "naik" (elevated) tanpa otorisasi yang sah.

Cara membedakan:
• Spoofing = memalsukan SIAPA kamu (identitas)
• Tampering = mengubah DATA/kode
• EoP = melakukan sesuatu yang TIDAK BOLEH kamu lakukan
• Repudiation = menyangkal bahwa kamu melakukan sesuatu
MFA Quiz 5

Manakah kombinasi berikut yang merupakan contoh BENAR dari Multi-Factor Authentication (MFA)?

✅ Jawaban: Password + OTP via SMS

MFA sejati mengharuskan minimal dua faktor dari KATEGORI BERBEDA:
• What you know: password, PIN, passphrase
• What you have: HP (OTP), token hardware, kartu
• What you are: fingerprint, face ID, iris

Password (know) + OTP ke HP (have) = 2 kategori berbeda = ✅ MFA
PIN + Password = 2× know = ❌ bukan MFA
Input Validation Quiz 6

Developer mendapatkan input tanggal lahir "30-02-2000" (30 Februari tidak ada). Validasi jenis apa yang gagal?

✅ Jawaban: Semantic validation

"30-02-2000" secara FORMAT (syntactic) bisa diterima: tiga angka dipisah tanda hubung. Tapi secara makna/logika (semantic): Februari tidak pernah punya 30 hari → gagal semantic validation.

Analogi:
• Syntactic: apakah kalimatnya pakai tata bahasa yang benar?
• Semantic: apakah kalimatnya masuk akal artinya?

"Colorless green ideas sleep furiously" → ✅ Syntactic (grammatically correct) ❌ Semantic (tidak masuk akal)
CSRF Quiz 7

User A sedang login di bank.com. Penyerang mengirimkan link ke A yang ketika diklik, secara diam-diam melakukan transfer. Apa yang membuat serangan ini berhasil?

✅ Jawaban: Browser otomatis sertakan cookie

Inilah inti dari CSRF (Cross-Site Request Forgery):
Browser selalu mengirimkan cookie yang relevan ke domain tujuan, TERMASUK jika request berasal dari situs lain. Penyerang tidak perlu tahu isi cookie – mereka hanya perlu membuat browser korban mengirimkan request ke target.

Mitigasi utama:
• Anti-CSRF token (server generate token unik, embed di form, verifikasi saat submit)
• SameSite=Strict cookie flag (browser tidak kirim cookie untuk cross-site request)
• Verifikasi Referrer/Origin header

Note: Opsi d adalah deskripsi XSS, bukan CSRF.
Attack Tree Quiz 4

Untuk mengeksfiltrasi data dari server, penyerang HARUS: (1) mendapatkan akses jaringan internal, DAN (2) bypass firewall database. Struktur node dalam attack tree?

✅ Jawaban: AND node

AND node digunakan ketika SEMUA kondisi/langkah anak harus terpenuhi untuk mencapai tujuan parent.

Kata kunci yang menandakan AND node: "harus", "dan", "both", "requires all"
Kata kunci yang menandakan OR node: "atau", "salah satu", "any of", "either"

Visual mental model:
AND node: parent ← [child1] ∧ [child2] (keduanya wajib)
OR node: parent ← [child1] ∨ [child2] (salah satu cukup)
XSS Quiz 7

Sebuah website menampilkan nama pengguna langsung dari database: <h1>Halo, <?= $username ?>!</h1>. Penyerang mendaftarkan dengan nama: <script>alert(1)</script>. Serangan apa ini dan bagaimana mitigasinya?

✅ Jawaban: Stored XSS

Ini adalah Stored (Persistent) XSS: script berbahaya disimpan ke database, dan dieksekusi setiap kali halaman dimuat oleh korban manapun.

Jenis XSS:
• Stored: payload tersimpan di database, dieksekusi setiap load
• Reflected: payload di URL, dieksekusi sekali saat klik link
• DOM-based: terjadi di client-side JavaScript

Mitigasi:
htmlspecialchars($username, ENT_QUOTES, 'UTF-8')
Mengubah < → &lt; dan > → &gt; sehingga tidak diinterpretasi sebagai HTML tag.

Content-Security-Policy juga membantu dengan membatasi sumber script yang diizinkan.
OAuth & Auth Quiz 5

Saat kamu login dengan Google di aplikasi pihak ketiga (misalnya Spotify), proses apa yang sebenarnya terjadi?

✅ Jawaban: Autentikasi + Otorisasi via Google

Alur OAuth 2.0 + OIDC:
1. Kamu klik "Login with Google" di Spotify
2. Redirect ke Google → autentikasi (verifikasi email + password)
3. Google minta konfirmasi: "Izinkan Spotify akses profil & email?"
4. Kamu setuju → Google beri authorization code ke Spotify
5. Spotify tukar code dengan access token + ID token
6. ID token berisi info siapa kamu → Spotify tahu kamu siapa

Spotify TIDAK PERNAH tahu password Google-mu. Ini prinsip inti OAuth: delegasi akses tanpa berbagi credential.
Security Headers Quiz 7

Pasangkan security header dengan fungsinya yang tepat. Header manakah yang KHUSUS melindungi dari serangan Clickjacking?

✅ Jawaban: X-Frame-Options: DENY

Clickjacking bekerja dengan menyembunyikan halaman target dalam iframe transparan di atas halaman palsu. Korban mengira klik tombol di halaman palsu, padahal kliknya "diteruskan" ke tombol di halaman target.

X-Frame-Options: DENY mencegah browser memuat halaman tersebut dalam frame/iframe apapun.

Alternatif modern: CSP dengan directive frame-ancestors 'none'

Ringkasan semua header:
• CSP → cegah XSS (whitelist script sources)
• HSTS → cegah MITM downgrade ke HTTP
• X-Frame-Options → cegah Clickjacking
• HttpOnly → cegah XSS steal cookie via JS
