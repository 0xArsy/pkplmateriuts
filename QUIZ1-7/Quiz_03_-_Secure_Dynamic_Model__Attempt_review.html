Conceptual Deep Dive

OCL: Precondition, Postcondition & @pre

pre:

Syarat yang HARUS benar SEBELUM operasi. Jika gagal → operasi DITOLAK. Postcondition tidak dievaluasi.

post:

Kondisi yang HARUS benar SETELAH operasi sukses. Jika tidak terpenuhi → postcondition dilanggar (bug).

@pre

Nilai atribut SEBELUM operasi dijalankan. Digunakan di postcondition untuk membandingkan perubahan.

TRICKY — Soal Quiz 3 Q3 (Salah di kuis!)

OCL: pre: password.size() >= 12. Password hanya 6 karakter → precondition GAGAL. TAPI, sistem tetap mengatur isAuthenticated = true → ini melanggar DUA hal: (1) Precondition dilanggar (harusnya operasi ditolak), (2) Postcondition dilanggar juga (karena sistem SEHARUSNYA menolak, bukan mengizinkan — state isAuthenticated=true tidak seharusnya terjadi dari operasi yang preconditionnya gagal). Jawaban: keduanya dilanggar.

Sequence Diagram — Elemen Penting

COMMON PITFALL

Stereotype dalam sequence diagram hanya anotasi desain — tidak menjamin implementasi keamanan. <> pada message tidak otomatis mengenkripsi message tersebut. Harus ada mekanisme nyata.

Activity Diagram — Elemen & Security Implications

SECURITY IMPLICATION

Jika error branch dimodelkan tapi tidak ada logging → sulit melakukan audit dan forensic analysis. Error tanpa logging = silent failure = inkonsistensi state yang tidak terdeteksi.

Secure Dynamic Modeling — Tujuan & Kelemahan Umum

Tujuan
  • • Mengidentifikasi kontrol keamanan yang hilang
  • • Mendeteksi potensi ancaman saat interaksi objek
  • • Menemukan operasi/class yang belum teridentifikasi (vs class diagram)
  • • Memformalkan perilaku yang diharapkan → membantu deteksi defect
Kelemahan Desain yang Umum Ditemukan
  • • Authentication SETELAH aksi kritis → kontrol akses terlambat
  • • Tidak ada error handling → silent failure & inkonsistensi state
  • • Stereotype tanpa constraint OCL → inkonsistensi annotasi vs implementasi
  • • Async call sebelum kondisi terpenuhi → precondition bisa gagal
  • • State diagram valid tapi sequence diagram melewati state → inkonsistensi

Algorithm & Logic Visualization

OCL — Cara Analisis Step-by-Step

// === SOAL QUIZ 3 Q3 — Yang Dijawab Salah ===
context Person::createLogin(username : String, password : String)
pre:  username.size() >= 8 and password.size() >= 12
post: self.isAuthenticated = true

// Skenario: password = "abc123" (hanya 6 karakter)
// Tapi sistem mengatur isAuthenticated = true

// ANALISIS:
// Step 1: Cek precondition
username.size() >= 8   → true  (misal username 10 karakter)
password.size() >= 12  → false (hanya 6 karakter!)
// PRE: true AND false = FALSE → PRECONDITION DILANGGAR ✗

// Step 2: Karena precondition gagal, operasi SEHARUSNYA ditolak
// TAPI sistem TETAP mengatur isAuthenticated = true
// Ini berarti operasi berjalan padahal seharusnya tidak!

// Step 3: Cek postcondition: self.isAuthenticated = true
// Postcondition HANYA valid jika precondition terpenuhi
// Karena precondition gagal, setting isAuthenticated=true dari
// operasi yang tidak valid = POSTCONDITION juga dilanggar

// JAWABAN: Pelanggaran PRECONDITION dan POSTCONDITION! ✓
MENTAL MODEL

Bayangkan OCL seperti kontrak: PRE = syarat masuk, POST = janji keluar. Jika syarat masuk tidak terpenuhi, kontrak batal total. Jika sistem tetap mengeksekusi + mengubah state → kontrak dilanggar dari kedua sisi.

Sequence Diagram — Authentication Order Matters

// ❌ SALAH: Authentication setelah aksi kritis
User ──→ System: requestTransfer(amount)
System ──→ Bank: processPayment(amount)   ← AKSI KRITIS terjadi!
System ──→ AuthService: verifyUser()      ← Auth baru dicek SETELAH!
// Masalah: kontrol akses diterapkan SETELAH aksi kritis terjadi
// Penyerang bisa memproses pembayaran sebelum diverifikasi!

// ✅ BENAR: Authentication sebelum aksi kritis
User ──→ System: requestTransfer(amount)
System ──→ AuthService: verifyUser()      ← Auth dicek DULU
AuthService ──→ System: [authenticated]
System ──→ Bank: processPayment(amount)   ← baru aksi kritis
// Urutan: Verify first, act second!

// ASYNCHRONOUS RISK:
// Jika confirm() dikirim sebelum payment.process() selesai:
// pre: self.payment.status = 'Success'
// Payment belum selesai → status masih 'Pending' → PRECONDITION GAGAL!

Activity Diagram — Node Reference

// Node Types dalam Activity Diagram:

// ● (lingkaran hitam penuh)   = Initial Node (start)
// ⊙ (lingkaran dengan titik)  = Activity Final Node (end)
// ◇ (diamond)                 = Decision Node (percabangan kondisi if/else)
// ── (horizontal bar tipis)   = Fork Node (mulai paralel)
// ══ (horizontal bar tebal)   = Join Node (sinkronisasi semua branch)

// Fork: satu alur menjadi BANYAK (paralel)
[Aktivitas A] ──→ [═══ FORK ═══] ──→ [Aktivitas B]
                       └──────────→ [Aktivitas C]
// B dan C berjalan bersamaan (paralel)

// Join: BANYAK alur menjadi satu (sinkronisasi)
[Aktivitas B] ──→ [═══ JOIN ═══] ──→ [Aktivitas E]
[Aktivitas C] ──→      ↑
// E hanya jalan setelah B DAN C selesai

// Security: Lock ticket sebelum booking = mencegah race condition!
// Tanpa lock: 2 user beli tiket terakhir bersamaan = double booking!
